## 1. 每日问题

### 问题 1：`fopen(filename, "a")` 和 `fopen(filename, "w")` 的区别是什么？

`"a"` 是 append 追加模式：

```text
保留旧内容，在文件末尾写入新记录。
```

`"w"` 是 write 覆盖模式：

```text
打开文件时先截断旧内容，再写入新内容。
```

对养殖记录系统来说，历史数据是资产。误用 `"w"` 会把之前的观测记录抹掉，后续复盘、趋势分析和模型训练都会失去数据基础。

### 问题 2：为什么这次不是编译错误，也不是链接错误？

这次代码能通过编译、链接，并能运行测试程序。

失败发生在运行阶段：

```text
csv_store_append_record写入行数测试未通过！
```

根因不是语法、类型或符号解析，而是文件写入行为不符合“追加两条记录”的预期，所以它属于：

```text
file_io / behavior regression
```

### 问题 3：为什么测试要写入两条记录并统计行数？

只写一条记录时，`"a"` 和 `"w"` 都可能看起来正常。

两次写入才能暴露区别：

```text
"a": 第一次写入 1 行，第二次追加后变 2 行。
"w": 第一次写入 1 行，第二次打开先清空，最终仍只剩 1 行。
```

所以存储层测试必须至少写入两条记录，并验证记录数量。

### 问题 4：`filename` 和 `FILE *fp` 的区别是什么？

`filename` 是文件路径字符串入口：

```c
const char *filename
```

它告诉程序“文件在哪里”。

`FILE *fp` 是 `fopen` 成功后返回的文件流操作入口：

```c
FILE *fp = fopen(filename, "a");
```

它让程序可以对这个文件执行 `fprintf`、`fclose` 等操作。

### 问题 5：为什么测试前要 `remove(filename)`？

因为追加模式会保留旧内容。

如果不先删除旧测试文件，上一次测试留下的记录会影响本次行数判断。

测试前清理文件，是为了让每次测试都从干净状态开始：

```text
remove(filename) -> 写入 record1 -> 写入 record2 -> 统计行数
```

### 问题 6：为什么 csv_store 不应该调用 judge/control？

csv_store 是持久化层，只保存原始记录。

边界应保持简单：

```text
record: 定义数据。
judge: 判断水质。
control: 生成建议。
csv_store: 保存原始记录。
```

如果 csv_store 调用 judge/control，存储层就会混入判断和决策逻辑，后续模块会越来越难测试、难替换。

## 2. 解答总结

今天的核心是把“能写文件”推进到“不会破坏历史记录”。

最重要的工程规则：

```text
养殖历史记录默认用追加模式 "a"。
涉及历史数据的测试，至少写入两条记录并验证记录数量。
```

这一步让项目从内存中的 `PondRecord` 走向可积累的外存记录。

## 4. 使用场景和启发思考

真实水产系统里，CSV 是第一版最小持久化层。

它的价值不是“高级”，而是：

```text
纯 C 标准库可实现。
容易打开观察。
容易测试。
后续可导入表格、Python、数据库或模型训练流程。
```

但 CSV 存储层必须保护历史记录。对养殖项目来说，覆盖写入不是小 bug，而是数据资产丢失。

## 5. 408 映射

数据结构：

- `PondRecord` 是内存中的结构化数据对象，csv_store 把它序列化成 CSV 文本记录。
- CSV 多行记录可以弱映射为外存上的顺序记录集合。

计算机组成原理：

- 本次 bug 不影响编译链接，失败发生在程序运行后的文件写入行为。
- `PondRecord` 在内存中是结构体，写入 CSV 后成为外存文本表示。

操作系统：

- `fopen / fprintf / fclose` 对应文件打开、写入和关闭。
- `"a"` 与 `"w"` 对应不同文件打开方式，直接影响文件内容是否保留。
- `filename` 定位路径，`FILE *fp` 表示已打开文件的操作入口。

计算机网络：

- 今日不涉及网络通信。
