## 1. 我遇到的问题

### 问题 1

- CSV 文件什么时候写表头？如果文件大小为 `0`，是不是就不写入？

### 解答

- 文件大小为 `0` 表示文件为空，不是不写入。
- 正确逻辑是：空文件先写 header，再写当前这条记录。
- CSV 表头本质上不是特殊结构，只是第一行普通文本，用来说明后面每列字段是什么。

### 闭环

- 用临时目录运行程序，第一次生成 `pond_records.csv`，检查第一行是否是 header。
- 第二次再运行，检查 header 是否没有重复。

### 使用场景和启发思考

- 真实采集系统需要让数据文件可读、可导入、可训练模型。
- 表头是数据合同的一部分，后续前端、后端、Python 分析脚本都依赖它。

### 问题 2

- `fopen(CSV_FILE_NAME, "a")` 里的 `"a"` 是什么？为什么是双引号字符串？

### 解答

- `"a"` 是传给 `fopen` 的打开模式字符串，意思是 append 追加模式。
- `fopen` 的第二个参数类型是字符串入口，可以接收 `"r"`、`"w"`、`"a"`、`"a+"`、`"rb"` 等模式，所以使用双引号。
- `'a'` 是单个字符，不是 `fopen` 需要的模式字符串。

### 闭环

- 口头区分：`CSV_FILE_NAME` 决定打开哪个文件，`"a"` 决定用什么方式打开。
- 运行验证：文件不存在时创建，文件存在时追加到末尾。

### 使用场景和启发思考

- 文件打开模式是存储层的第一道合同：追加、覆盖、只读会直接决定数据是否安全。

### 问题 3

- `FILE *fp`、`fseek(fp, 0, SEEK_END)`、`ftell(fp)` 到底是什么？为什么移动到末尾后当前位置就是文件大小？

### 解答

- `FILE *fp` 不是文件本身，而是 C 标准库给程序的文件操作入口。
- `fseek(fp, 0, SEEK_END)` 是把文件位置指针移动到文件末尾。
- `ftell(fp)` 是询问当前位置距离文件开头有多少字节。
- 当位置已经在末尾时，“开头到当前位置的距离”就是整个文件大小。

### 闭环

- 用书签类比：文件开头是第 0 字节，书签移动到最后一页，书签距离第一页的距离就是文件长度。
- 在 `save_pond_record_csv()` 中用 `ftell(fp) == 0` 判断空文件。

### 使用场景和启发思考

- 这是后续日志、缓存、二进制文件、断点续写、ring buffer 的前置模型。

### 问题 4

- 程序运行后为什么看不到 CSV 文件？

### 解答

- `pond_records.csv` 是按当前工作目录生成的，不一定在 `exercises/06-projects/` 里。
- 运行 `make 61` 时，程序从仓库根目录执行，所以 CSV 会生成在仓库根目录。
- 如果已有旧 CSV 文件，`file_size == 0` 不成立，所以不会补表头。

### 闭环

- 用 `find . -name 'pond_records.csv'` 找真实位置。
- 用临时空目录验证新文件是否自动带 header。

### 使用场景和启发思考

- 文件路径是工程问题，不只是 C 语法问题。后续要区分源码目录、运行目录、数据目录和测试目录。

### 补充问题 2

- 为什么 SRS 不应该只按日期复习，还要有严格标签和相关标签图？

### 解答

- 只按日期会让旧问题平均涌入，容易吞掉当天主线。
- 严格标签能定位单个知识点，例如 `FILE *`、`CSV`、`结构体指针`。
- 相关标签图能把一个卡点连接到相邻知识点，例如 `FILE *` 关联 `fopen`、`fprintf`、`fseek`、`ftell`、`文件入口`。
- 主线权重能让 `06-projects` 当前真正相关的问题优先出现。

### 闭环

- 使用 `python3 daily/daily_review.py map` 查看标签地图。
- 使用 `python3 daily/daily_review.py list --minutes 60 --related-tag "FILE *"` 做精准复习。

### 使用场景和启发思考

- 这是把问题笔记升级成“学习调度系统”，让复习服务项目推进，而不是拖慢项目推进。
