# 2026-05-11 系统动力学、反馈告警与参考项目定位

## 我今天遇到的问题

### 1. 系统动力学草图让实现变复杂

问题：

`2.c` 拆出太多系统动力学结构体后，我反而不清楚今天该推进哪里。

解答：

系统动力学是建模工具，不是提前复杂化实现。当前应把 `2.c` 当草图，把主线放回 `1.c` 的最小闭环。

闭环：

新增功能前先判断它属于状态、关系、反馈、观察、记忆还是控制。

使用场景和启发思考：

架构图应该降低复杂度；如果它让人迷路，就先回到最小可运行系统。

### 2. 函数作用域边界不清

问题：

我在函数里直接写 `pond.stock.oxygen`，但 `pond` 是 `main` 里的局部变量。

解答：

函数只能直接使用自己的参数和局部变量。传入 `stock` 和 `params` 后，应使用 `stock.oxygen < params.oxygen_low_limit`。

闭环：

写函数体前先读函数签名。

使用场景和启发思考：

函数参数就是模块边界。真实工程中，边界越清楚，代码越可维护。

### 3. 判断层和显示层混在一起

问题：

我一开始在 `print_aeration_action()` 中重新写低氧判断。

解答：

`needs_aeration()` 负责判断；显示、告警、存储都应复用它，避免规则分裂。

闭环：

```text
needs_aeration() -> 判断
print_aeration_action() -> 动作显示
print_oxygen_alert() -> 告警显示
save_oxygen_alert_csv() -> 告警记忆
```

使用场景和启发思考：

真实系统里，同一条规则应只有一个源头。

### 4. 可解释输出和 `printf` 参数边界

问题：

我需要让输出说明触发原因，并且曾把 `printf` 参数写到括号外。

解答：

可解释输出要显示塘口、当前值、阈值和判断关系；`printf` 的格式串和参数必须在同一组括号里。

闭环：

检查输出时同时确认两件事：解释文字是否符合判断路径，`%` 占位符是否与参数匹配。

使用场景和启发思考：

判断正确不代表解释正确。可解释输出会延伸到告警、前端和论文表达。

### 5. `return 1` 与副作用的关系

问题：

我不理解为什么无告警时 `save_oxygen_alert_csv()` 也返回 `1`，以及 `if (!save...)` 正常不输出。

解答：

没有告警就是正常跳过，不是失败。`if (!save...)` 是异常处理，只有返回 `0` 才打印错误。

闭环：

```text
无告警 -> return 1
保存成功 -> return 1
保存失败 -> return 0
```

使用场景和启发思考：

返回值用于报告函数是否完成职责；副作用才是写文件、打印、控制设备等外部动作。

问题：

每天推进前先定位系统层，再写最小闭环代码。

## 补充的高质量问题

### 1. 告警 CSV 是否应该写表头

问题：

告警 CSV 是日志还是数据集？

解答：

临时日志可以无 header；后续给 Python、图表或模型训练用时，应写 header。

闭环：

升级 `save_oxygen_alert_csv()`：空文件先写 header，再写事件记录。

使用场景和启发思考：

数据一旦要被机器消费，就要按数据集而不是随手日志来设计。

### 2. 何时进入多文件模块化

问题：

`1.c` 已有多个职责，是否马上拆文件？

解答：

先收口 Stage 1，再进入 Stage 2。拆文件应发生在职责边界稳定之后。

闭环：

先清理运行产物、旧注释，并能说明每个函数属于哪一层。

使用场景和启发思考：

工程化不是提前拆，而是把稳定边界落成结构。
