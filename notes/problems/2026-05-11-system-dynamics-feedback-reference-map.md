# 2026-05-11 系统动力学、反馈告警与参考项目定位

## 我今天遇到的问题

### 1. `2.c` 结构体太多导致混乱

问题：

`2.c` 用系统动力学拆出了很多结构体，但我反而不清楚今天该推进哪里。

解答：

系统动力学是建模工具，不是提前堆结构体。当前阶段应把 `2.c` 当草图，把主线放回更可控的 `1.c`。

闭环：

新增功能前先判断它属于状态、关系、反馈、观察、记忆还是控制。

使用场景和启发思考：

真实工程里，架构图应该降低复杂度。如果模型本身让人迷路，就先回到最小可运行闭环。

### 2. 函数内部为什么不能直接用 `main` 里的 `pond`

问题：

在 `judge_pond_feedback(...)` 里写 `pond.stock.oxygen`，编译报错 `use of undeclared identifier 'pond'`。

解答：

函数只能直接访问自己的参数和局部变量。`pond` 是 `main` 的局部变量，没有传入函数，所以函数看不到它。

闭环：

应使用函数参数：

```c
stock.oxygen < params.oxygen_low_limit
```

使用场景和启发思考：

函数边界就是作用域边界。以后写函数体前，先看函数签名中给了哪些信息。

### 3. 为什么显示函数要复用 `needs_aeration(record)`

问题：

我一开始在显示函数里重新写判断条件，后来改成复用 `needs_aeration(record)`。

解答：

`needs_aeration()` 属于判断层，`print_aeration_action()` 属于显示层。判断规则集中后，未来修改阈值逻辑不会造成多处条件分裂。

闭环：

形成分层链路：

```text
needs_aeration() -> 判断
print_aeration_action() -> 动作显示
print_oxygen_alert() -> 告警显示
save_oxygen_alert_csv() -> 告警记忆
```

使用场景和启发思考：

真实系统里，显示、存储、控制都应该消费同一份判断结果，而不是各自重新判断。

### 4. 为什么可解释输出要显示触发原因

问题：

只显示 `aerator ON/OFF` 不够，还要显示 `oxygen 4.2 < 5.0` 这样的判断依据。

解答：

人看到动作建议后需要知道塘口、当前值、阈值和触发原因。可解释输出是把判断逻辑显式暴露给使用者。

闭环：

正常氧分支必须显示：

```text
oxygen >= OXYGEN_LOW_LIMIT
```

使用场景和启发思考：

这会延伸到云端告警、手机推送、论文里的可解释控制。判断正确不代表解释文字正确。

### 5. `printf` 换行写参数为什么报错

问题：

我把 `printf` 的格式字符串和参数写断了，导致占位符找不到参数。

解答：

格式字符串和所有数据参数必须在同一组 `printf(...)` 括号里。

闭环：

写成：

```c
printf("Alert: Pond %c low oxygen detected (oxygen %.1f < %.1f)\n",
       record.pond_id, record.oxygen, OXYGEN_LOW_LIMIT);
```

使用场景和启发思考：

以后检查 `printf` / `fprintf` / `scanf` 时，先数 `%`，再核对同一组括号里的参数数量和类型。

### 6. `save_oxygen_alert_csv()` 为什么有两个 `return 1`

问题：

我理解第一个 `return 1` 是没有告警正常跳过，第二个 `return 1` 是保存后返回，但不理解为什么没保存也算成功。

解答：

函数职责是处理告警保存流程，不是每次都必须写文件。没有告警时没有东西需要保存，也算正常完成。

闭环：

```text
没有告警 -> return 1
有告警且保存成功 -> return 1
有告警但文件打不开 -> return 0
```

使用场景和启发思考：

成功不是“发生写入”，而是“函数完成自己的职责”。这会用于上传、控制、缓存等场景。

### 7. `if (!save_oxygen_alert_csv(record))` 为什么正常不输出

问题：

这段代码大多数时候不打印，因为返回值通常是 `1`。

解答：

它是异常处理，不是成功提示。返回 `1` 表示正常，`!1` 为假；返回 `0` 表示失败，`!0` 为真。

闭环：

低氧时副作用是写入告警 CSV；正常氧时无告警记录，但仍然正常完成。

使用场景和启发思考：

返回值和副作用要分开理解。终端没输出，不代表函数没有做事。

### 8. 宏名为什么改成 `OXYGEN_ALERT_FILE_NAME`

问题：

我新增了告警文件名宏，后来需要把名字改得更明确。

解答：

`OXYGEN_ALERT` 像事件或状态，`OXYGEN_ALERT_FILE_NAME` 明确说明它是文件名配置。

闭环：

把硬编码文件名收敛为集中配置：

```c
fopen(OXYGEN_ALERT_FILE_NAME, "a")
```

使用场景和启发思考：

命名要表达系统角色。阈值、文件名、采样间隔、设备编号都应逐步集中配置。

### 9. 为什么需要完整参考项目

问题：

我需要一个完整水产养殖项目作为教学地图，否则不清楚当前 `1.c` 在整个系统里的位置。

解答：

`1.c` 是训练场，适合逐步闭环；`C-Journey-Framework/reference/aqualog-reference` 是地图，用来定位未来形态，不是直接照抄。

闭环：

今天建立双语教学地图和可运行参考项目，用于对照：

```text
record / judge / output / csv_store / sensor_sim / control / stats / tests
```

使用场景和启发思考：

以后每天推进前先问：我今天做的是参考项目里的哪一层？

## 你补充的高质量问题

### 1. 告警 CSV 是否应该写表头

问题：

告警 CSV 当前像事件日志一样追加，是否需要 header？

解答：

如果只是临时日志，可以不写；如果后续给 Python、图表或模型训练使用，最好写 header。

闭环：

后续升级 `save_oxygen_alert_csv()`：空文件先写 header，再写告警事件。

使用场景和启发思考：

数据一旦要被机器消费，就应该更像数据集，而不只是日志。

### 2. `2.c` 是否保留

问题：

`2.c` 当前不是主线，是否还值得保留？

解答：

值得保留。它是系统动力学草图，不作为当前实现主线。

闭环：

进入多文件模块化时，再吸收 `2.c` 的系统边界思想。

使用场景和启发思考：

工程里草图和实现可以分离。草图负责方向，实现负责闭环。

### 3. 什么时候进入多文件模块化

问题：

`1.c` 已经有多个职责，是否应该马上拆文件？

解答：

还不急。先确认 Stage 1 稳定，再进入 Stage 2。

闭环：

进入 Stage 2 前先完成：

```text
1. 运行产物边界清楚
2. 旧注释收干净
3. 能说明每个函数属于哪一层
```

使用场景和启发思考：

拆文件不是为了显得工程化，而是当职责边界已经稳定时，把边界落成文件结构。
